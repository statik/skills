01 Design thesis
Design thesis

Make design memos feel like reviewable presentations

The design-memo skill makes self-contained HTML memos the default output for non-trivial plans, architecture proposals, implementation approaches, decision records, and review documents.

Decision Default behavior Why it matters
Output format Standalone HTML Reviewers can scan, navigate, and share a memo without reading the chat transcript.
Simple plans Ask first A two-line fix should not be forced into a full presentation.
Visual system Tufte-inspired editorial layout with lightweight scrollytelling Dense implementation detail stays readable while still feeling like a presentation.
This memo is now a proof of the scrollytelling template, not just a long-form page with a progress bar.

Template behavior

The memo behaves like a guided scroll

Progress bar Fixed at the top and exposed as a named ARIA progressbar.
Section rail Keyboard-focusable dots are generated from each section and show hover/focus labels.
Chapter indicator Fixed bottom-left, updates the section number and eyebrow label as the reader scrolls.
Active cards Cards lift and brighten as they become active; no-JS readers still get fully readable content.
The key shift from the old memo is the slide/card rhythm: each section gets a full-height stage and a focused review unit.
Design decisions

The skill keeps an escape hatch for lightweight work

Trigger scope Use the skill for plans, designs, proposals, implementation approaches, decision records, and review documents.
Simple plans Ask whether plain text would be better for quick fixes or single-step tasks.
Output location Prefer an existing docs/ directory; ask before creating one.
Plugin metadata Keep plugin.json name stable and broaden only the description.
The default is HTML for non-trivial work, not HTML for every response. The skill still requires judgment.
Skill structure

The instruction set now describes both content and presentation

Area What the skill specifies
Purpose and output Standalone HTML memos with inline CSS/JS and enough implementation detail for review.
Visual direction Tufte-inspired editorial layout with an Arrival-inspired palette.
Scrollytelling Progress, section rail, chapter indicator, active-card transitions, and small-screen simplification.
Accessibility WCAG contrast, keyboard dots, ARIA progress, active-step state, no-JS fallback, and reduced motion.
Review quality Explicit rule ordering, validation matrices, source footers, caveats, and unknown/default behavior.
HTML skeleton Cards, grids, source blocks, badges, callouts, code blocks, semantic landmarks, and generated navigation.

Accessibility

The richer presentation keeps the accessibility contract

Requirement Implementation
Progress role="progressbar", named label, and updated aria-valuenow.
Keyboard navigation Rail dots are real <button> elements with visible focus rings and section number shortcuts.
Presentation shortcuts ArrowRight, ArrowDown, PageDown, and Space advance; ArrowLeft, ArrowUp, and PageUp go back; 1-9 jump to sections.
Current section Only the active dot receives aria-current="step"; inactive dots omit the attribute.
No JavaScript Content remains visible by default; dimming is scoped to the generated .js class.
Reduced motion Transitions, smooth scrolling, and scroll snapping are disabled for reduced-motion users.
Scope boundaries

What was intentionally left out

install.sh Already skill-agnostic; it accepts the skill name as an argument.
release.yml Still hardcoded around dns-troubleshooter. A future PR should make releases multi-skill-aware.

The release workflow is the main follow-up when this skill needs independent versioned releases.
Confidence and caveats

The branch is ready with one known operational gap

High confidence The skill validates, follows the repository patterns, and this memo exercises the richer HTML scaffold.
Medium confidence The trigger scope depends on agent judgment for the line between "quick plan" and "non-trivial memo".
Known gap There are no automated evals for presentation quality yet.
Follow-up Parameterize or split release.yml before publishing multiple independently versioned skills.
